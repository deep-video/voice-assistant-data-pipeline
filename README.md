## 项目背景

### 核心挑战

1. **缺乏真实数据**：项目初期没有真实用户语音数据
2. **数据真实性差**：早期生成的数据过于简单、模板化
3. **意图边界模糊**：相似意图之间容易混淆（如音乐推荐 vs 媒体推荐）
4. **多样性不足**：句式单一，过度依赖"推荐"、"来点"等固定模式

### 目标

生成 1000+ 条/意图 的高质量训练数据，覆盖 13 个意图类别，包含：
- 种子数据（20%）
- 扩展数据（70%）
- 边界样本（10%）

## 架构演进

### 第一阶段：LLM 直接生成

**初始思路**：让 LLM 一次性生成所有训练数据

**尝试路径**：

#### 尝试 1：单次调用生成 100 条
```python
# 一次性让模型生成 100 条
seeds = generate_seeds(num_seeds=100)
```

**问题**：
- ❌ 句式模板化严重（"来点..."占 30%+）
- ❌ 类型分布不均（模糊需求 54%、直接请求 41%）
- ❌ 过度使用"推荐"一词
- ❌ **相似度过高**：100 条样本内部相似度太高，缺乏多样性

#### 尝试 2：分批并发 + 扰动策略
```python
# 分 5 批并发，每批 30 条，添加随机扰动
batch_size = 30
num_batches = 5
```

**发现**：
- ✅ "其他"句式从 40% 提升到 57%
- ✅ "来点"句式从 30% 降至 21%
- ❌ **根本问题未解决**：批次生成仍然无法保证多样性，因为每批之间仍然会重复

#### 尝试 3：场景矩阵 + Few-shot

**设计思路**：让 LLM 自动生成场景矩阵（时间 × 地点 × 情绪 × 行为），然后以场景作为 few-shot 示例生成样本，再基于场景进行泛化

```python
# LLM 自动生成场景矩阵
scenes = [
    {"time": "早上", "location": "上班路上", "mood": "匆忙"},
    {"time": "晚上", "location": "停车场", "mood": "放松"},
    ...
]
# 以场景作为 few-shot 示例生成样本
# 然后基于场景进行泛化
```

**与种子生成方案的关系**：
> 场景矩阵方案与种子生成方案是**并列的两种方案**，不是在种子生成基础上的改进。两者都试图解决"单次生成大量样本相似度高"的问题，但采用了不同的路径。

**问题**：
- ❌ **数据生硬**：基于场景矩阵生成的数据过于刻意，不自然
- ❌ **泛化效果差**：在场景上泛化（如"上班路上"→"下班回家"）对意图识别模型帮助有限
- ❌ **覆盖有限**：场景组合无法覆盖真实世界的多样性
- ❌ **对比劣势**：在场景上泛化 vs 在种子上泛化，后者效果更好

**关键洞察**：
1. **场景泛化无意义**：用户说"上班路上想听歌"和"下班回家想听歌"，在意图识别模型看来是**同一个样本**（都是 music_recommendation）
2. **句式变化才是关键**："推荐几首歌"和"有歌推荐没"虽然语义相同，但句式差异能帮助模型学习不同的表达方式
3. **种子 - 扩展更优**：在种子上泛化（句式变化、口语噪声）比场景泛化效果更好

**解决方案**：转向**种子 - 扩展架构**

### 第二阶段：种子 - 扩展架构（当前方案）

#### 核心设计思想

**两阶段生成**：
1. **种子生成**：LLM 并发生成少量高质量种子（30 条/批 × 多批）
2. **种子扩展**：基于每个种子生成 8-10 个变体

**为什么这样设计？**

| 设计决策 | 原因 |
|---------|------|
| **种子阶段并发生成** | 单次生成 100 条效果差，30 条是实验得出的最佳平衡点 |
| **多批次并发** | 每批使用不同的 temperature 和提示词侧重，保证批次间多样性 |
| **种子扩展** | 基于种子变体可在保持质量的前提下快速扩大规模 |
| **句式变化优先** | 实验发现"换场景"等名词变化无意义，句式变化才是关键 |

#### 种子生成策略

**并发批次设计**：
```python
# 每批 30 条，多批并发
batch_size = 30  # 实验得出的最佳数字
num_batches = 5  # 根据目标数量动态计算

# 每批不同的扰动参数
for batch in range(num_batches):
    temperature = random.choice([0.7, 0.8, 0.9, 1.0])
    emphasis = random.choice([
        "侧重直接请求（40%）",
        "侧重情绪驱动（30%）",
        "侧重倒装句（20%）",
        "侧重省略句（15%）",
    ])
```

**为什么是 30 条？**
- < 20 条：效率太低，需要更多批次
- > 50 条：质量下降，后半部分重复率高
- **30 条**：质量和效率的最佳平衡点

#### 扩展策略优化

**关键改进**：
1. **过采样去重**：生成 1.5 倍数据，去重后保留目标数量
2. **变体策略**（按重要性排序）：
   - **句式变化（60%）**：倒装、省略、语气词、语序调整
     - 例："推荐几首歌" → "歌的话，有推荐的没" → "有歌推荐没"
   - **口语噪声（20%）**：填充词、重复、纠错
     - 例："嗯...那个...推荐一首歌" / "推荐一首歌，不对，来首歌"
   - **场景泛化（20%）**：不同驾驶场景

3. **批次并发**：4 个 worker 并发生成，提升效率

**为什么句式变化比场景变化重要？**
> 实验发现：用户说"上班路上想听歌"和"下班回家想听歌"，在意图识别模型看来是**同一个样本**（都是 music_recommendation）。但"推荐几首歌"和"有歌推荐没"虽然语义相同，**句式差异能帮助模型学习不同的表达方式**。

### 第三阶段：边界样本与质量评估

#### 边界样本自动生成

**问题**：13 个意图之间存在边界模糊情况，需要专门的边界样本

**解决方案**：
1. **自动识别混淆意图**：让 LLM 分析每个意图可能混淆的其他意图
2. **针对性生成边界样本**：针对每对混淆意图生成易混淆样本

```python

    # 输出格式：[{"intent": "xxx", "reason": "混淆原因"}]
```

**示例**：`music_recommendation` 的混淆意图
- `media_recommendation`：用户说"推荐点好看的内容"——"内容"指视频/播客而不是音乐
- `history_query`：用户说"刚才那首歌是什么"——是历史记录，非新的音乐推荐请求
- `car_control`：用户说"把音乐打开"——"打开"为控制车辆音响系统而非推荐音乐

**生成策略**：
- 自动比例分配：根据配置的边界数据占比（默认 10%）自动计算每个混淆意图对需要生成的样本数
- 动态计算：例如目标 1000 条数据，边界占比 10%，共需 100 条边界样本，平均分配到各混淆意图对

#### 质量评估策略

**评估挑战**：
- 评分机制（0-5 分）难以定义：有些非清晰意图的样本是人工特意生成的边界样本，这种样本是需要的，无法简单打分
- 二分类（保留/丢弃）效果比打分更好

**当前方案**：
- 使用二分类判断：模型输出"保留"或"丢弃"并给出理由
- 评估维度：
  1. 是否包含与意图相关的隐含词（如音乐意图需包含"听"、"歌"、"旋律"等）
  2. 表达是否自然，像真人对语音助手说话
  3. 长度是否在 5-20 字范围内


**评估模型**：Qwen3-1.7B（本地部署）

### 效果对比与局限性

#### 模型表现变化

| 阶段 | 评估模型 | Precision/Recall/F1 | 说明 |
|------|---------|---------------------|------|
| 初始方案（单次生成） | Qwen3-1.7B | 90%+ | 数据简单，模型容易学习 |
| 当前方案（种子 - 扩展 + 边界） | Qwen3-1.7B | 70%+ | 数据复杂度提升，边界样本增加难度 |

**指标下降的原因**：
1. **数据复杂度提升**：从单一意图样本到包含边界样本、噪声样本
2. **边界样本引入**：故意设计易混淆样本，增加分类难度
3. **口语噪声**：添加填充词、重复、纠错等真实场景特征
4. **句式多样性**：不再是模板化表达，模型需要学习更本质的特征

**积极意义**：
> 指标下降**不代表质量下降**，而是数据更接近真实场景的复杂性。90%+ 的准确率是因为数据太简单，70% 的准确率在更复杂的数据上反而更有价值。

#### 局限性：相似样本检测

**问题**：无法保证生成数据中不出现相似样本，只能通过扩大数据生成规模来缓解。

**尝试思路（未实现）**：

将 100 个样本同时输入模型，让模型判断哪些样本是相似的，输出相似样本的编号对（如 12 和 45 相似、78 和 92 相似）。这种方案理论上可以一次性检测大批量样本的相似关系，但实际操作中面临成本、组合数和准确率等多重问题。

**为什么未实现**：
1. **成本过高**：
   - **批次间成本**：完整数据集需要分多批处理（如 1000 条数据需要 10 批），每批都需要调用大模型
   - **交叉验证成本**：不同批次之间还需要相互判断（如第 1 批的样本与第 2 批的样本是否相似），导致调用次数呈指数级增长
2. **准确率与成本的矛盾**：
   - **大样本量（如 100 条同时输入）**：需要模型同时判断 100 条样本之间的相似关系，组合太多导致准确率很低
   - **小样本量（如每次 30 条样本）**：准确率可能提升，但需要调用更多次 API，成本进一步扩大
3. **定义模糊**："相似"的定义本身需要人工判断（句式相似？语义相似？字面上相似？）

**当前策略**：
> **接受相似样本的存在**而非尝试检测。由于相似样本检测成本高且准确率低，实际上无法有效识别哪些是相似样本。因此选择直接扩大生成规模，即使存在一定比例的相似样本也无法检测和剔除，只能接受。

#### 其他局限性：困难样本的副作用

**问题**：为了生成边界和困难样本，可能会产生一些意图不正确的数据

**原因**：
- 边界样本故意设计得模糊，可能导致模型生成偏离原意图的句子
- 口语噪声的添加可能改变句子的原始意图
- 句式变换过度可能导致语义扭曲

**解决方案**：
1. **质量评估阶段筛选**：在质量过滤环节，对意图不明确的样本进行剔除或修正
2. **评估阶段修正**：在意图分类评估中，发现标注意图与模型预测不一致的样本，人工审核并修正
3. **接受一定错误率**：认识到困难样本生成必然伴随一定比例的"错误"，这是为了提升模型鲁棒性必须付出的代价

**未来方向**：
- 探索更智能的质量筛选方法（如多轮评估、人工审核）
- 在保持数据多样性的同时，平衡意图的准确性

**为什么不推荐 Embedding/BM25 方法**：
- 实际测试效果较差：Embedding 模型和 BM25 都不适合检测句式变换的相似样本
- 语义相似是预期行为：我们想要的是句式变换（如"推荐几首歌"→"有歌推荐没"），这些样本天然语义相似
- 向量模型不可靠：通过向量模型找相似样本，可能会错误地剔除我们需要的句式变换样本

## 当前架构

### 完整 Pipeline 流程

```
┌─────────────────────────────────────────────────────────────┐
│                     数据生成 Pipeline                        │
├─────────────────────────────────────────────────────────────┤
│  步骤 1: 种子生成                                            │
│  - 并发生成：batch_size=30（实验最佳值）                     │
│  - 多批次并发：每批不同的 temperature 和句式侧重              │
│  - 去重保存：过采样 1.5 倍，去重后保留目标数量                 │
│  - 输出：intent_seeds.json                                   │
├─────────────────────────────────────────────────────────────┤
│  步骤 2: 种子扩展                                            │
│  - 变体策略：句式变化（60%）+ 口语噪声（20%）+ 场景（20%）   │
│  - 扩展倍数：每个种子生成 8-10 个变体                          │
│  - 过采样去重：生成 1.5 倍，去重后保留                         │
│  - 输出：intent_expanded.json                                │
├─────────────────────────────────────────────────────────────┤
│  步骤 3: 边界样本生成                                        │
│  - 自动识别混淆意图：LLM 分析可能混淆的意图对                 │
│  - 自动比例分配：根据配置的边界占比自动计算每个混淆意图对的样本数 │
│  - 输出：intent_vs_confusing.json                            │
├─────────────────────────────────────────────────────────────┤
│  步骤 4: 质量过滤                                            │
│  - 评估模型：Qwen3-1.7B（本地部署）                          │
│  - 二分类判断：保留/丢弃（而非打分）                          │
│  - 评估维度：隐含词、自然度、长度                            │
│  - 输出：intent_keep.json / intent_discard.json              │
├─────────────────────────────────────────────────────────────┤
│  步骤 5: 数据合并                                            │
│  - 合并所有意图的保留数据                                     │
│  - 添加 intent 字段                                           │
│  - 输出：all_intents_dataset.json                            │
└─────────────────────────────────────────────────────────────┘
```

### 数据比例

```python
# PipelineConfig 配置
SEED_RATIO = 0.20      # 种子占比 20%
EXPAND_RATIO = 0.70    # 扩展占比 70%
BOUNDARY_RATIO = 0.10  # 边界占比 10%

```

**以目标 1000 条为例**：
- 需要生成总量：1000 / 0.85 ≈ 1176 条
- 种子数据：1176 × 0.20 ≈ 235 条
- 扩展数据：1176 × 0.70 ≈ 823 条
- 边界数据：1176 × 0.10 ≈ 118 条

## 快速开始

### 环境准备

```bash
# 配置 API（在 src/config/api_config.py 中）
QWEN_API_KEY = "your_api_key"
QWEN_API_URL = "https://dashscope.aliyuncs.com/compatible-mode/v1"
QWEN_MODEL_NAME = "qwen-plus"
```

### 生成数据

```bash
# 生成所有意图的数据（默认每个意图 1000 条）

# 生成指定意图的数据
python src/core/pipeline_manager.py --intent music_recommendation --target 1000

python src/core/pipeline_manager.py --target 100 --no-eval
```

- `test_predictions.json`：测试集详细预测结果
- `test_metrics.txt`：各意图的 precision、recall、F1、FPR 指标

## 评估指标

### 意图分类评估

使用 Qwen3-1.7B 作为评估模型，对测试集进行意图预测：

**指标说明**：
- **Precision**：预测为该意图的样本中，实际为该意图的比例
- **Recall**：实际为该意图的样本中，被正确预测的比例
- **F1 Score**：Precision 和 Recall 的调和平均
- **FPR**：实际不为该意图的样本中，被错误预测为该意图的比例

**典型结果**（示例）：
```
music_recommendation:
  样本数 (Support): 204
  Precision:        0.7026
  Recall:           0.6716
  F1 Score:         0.6867
  FPR:              0.0275

```

## 关键设计决策总结

### 为什么使用 LLM 而非规则？

✅ **LLM 优势**：
- 生成自然、口语化的表达
- 自动处理语言多样性
- 无需手动维护规则模板

❌ **规则劣势**：
- 规则有限，覆盖不全
- 随机组合不通顺
- 维护成本高

### 为什么放弃场景矩阵 + Few-shot 方案？

❌ **场景矩阵问题**：
- 数据生硬：基于矩阵组合生成的数据过于刻意
- 成本高昂：需要人工设计场景维度
- 覆盖有限：场景组合无法覆盖真实世界

✅ **当前方案**：让 LLM 直接考虑"真实场景中用户可能的输出"

### 为什么分种子和扩展两阶段？

**尝试过的方案**：
- ❌ 单次生成 100 条：相似度高，后半部分质量下降
- ❌ 分批生成：批次间仍然重复，"换汤不换药"
- ❌ 场景矩阵 + Few-shot：数据生硬，不自然

✅ **种子 - 扩展方案**：
- **种子阶段**：并发批次生成（30 条/批），保证多样性和质量
- **扩展阶段**：基于种子变体，句式变化比场景变化更有意义
- **质量过滤**：双重保障，剔除不合格样本

**关键洞察**：
> 实验发现：单次生成 100 条效果差，30 条是实验得出的最佳平衡点。句式变化（如"推荐几首歌"→"有歌推荐没"）比场景变化（如"上班路上"→"下班回家"）对模型训练更有价值。

### 为什么边界样本需要自动生成？

- 意图边界模糊是分类难点
- 人工定义混淆意图主观且耗时
- LLM 能发现人类忽略的边界情况
- 针对性生成边界样本，提升模型判别能力

### 为什么使用并发策略？

- 单批次生成容易模板化
- 多批次 + 扰动（temperature、句式侧重）提升多样性
- 线程池并发提升效率
- **30 条/批**：实验得出的最佳平衡点（<20 效率低，>50 质量降）

### 为什么不精确检测相似样本？

**困难**：
- 组合爆炸：100 条样本有 C(100,2)=4950 种组合
- 成本高昂：需要多次调用模型检测
- 定义模糊："相似"的定义需要人工判断
- 准确率不确定

✅ **当前策略**：接受相似样本的存在而非尝试检测
> 由于相似样本检测成本高且准确率低，实际上无法有效识别哪些是相似样本。因此选择直接扩大生成规模，即使存在一定比例的相似样本也无法检测和剔除，只能接受这一现实。

### 为什么模型指标从 90%+ 下降到 70%？

**原因**：
1. 数据复杂度提升：从单一意图到边界样本、噪声样本
2. 边界样本引入：故意设计易混淆样本，增加难度
3. 口语噪声：添加填充词、重复等真实场景特征
4. 句式多样性：不再是模板化表达

**积极意义**：
> 指标下降**不代表质量下降**，而是数据更接近真实场景。90%+ 可能因为数据太简单，70% 在更复杂数据上反而更有价值。
